## 2. Obsidian Plugin Improvements

### High-Priority Additions (Install These)

#### Linter
**What it does:** Auto-formats markdown on save -- consistent heading spacing, trailing whitespace, YAML frontmatter formatting, ordered list numbering.

**Why TFM needs it:** With 10 people eventually editing client files, markdown inconsistency will creep in fast. Dataview queries break when frontmatter is malformed. The 25 client intelligence files all need identical structure for Claude Code to parse them reliably. Linter enforces this without anyone thinking about it.

**Specific config to set:**
- Enable YAML sort and formatting
- Enable heading blank lines
- Enable trailing whitespace removal
- Disable any rules that would reformat tables (Advanced Tables handles that)

#### Kanban
**What it does:** Turns a markdown file into a drag-and-drop board view.

**Why TFM needs it:** The creative pipeline is currently tracked in Notion, but having a local Kanban per client (or one master board) would let GMs see `Concept -> In Production -> In Review -> Live -> Killed` at a glance without leaving Obsidian. Each card is just a markdown link, so it stays searchable and Claude Code can read it.

**Suggested boards:**
- `dashboards/creative-pipeline.md` -- master Kanban with lanes: Scripted, In Design, In Review, QA'd, Live, Paused, Killed
- Per-client Kanbans optional for high-volume clients like MarketBeat or Contrarian Thinking

#### Tag Wrangler
**What it does:** Rename, merge, delete, and search tags from the tag pane. Right-click context menu on tags.

**Why TFM needs it:** With 25 clients and multiple file types (meeting notes, reports, intelligence files, research), tags will proliferate. Tag Wrangler lets you bulk-rename `#biweekly` to `#bi-weekly` or merge `#report` and `#friday-report` without editing 50 files by hand. Essential as the team grows.

#### Commander
**What it does:** Add any command to the ribbon, title bar, status bar, or right-click menu. Custom hotkeys with icons.

**Why TFM needs it:** Create one-click buttons for the team's most common actions:
- "New Friday Report" (triggers Templater)
- "New Meeting Notes" (triggers Templater)
- "Open Client Portfolio Dashboard"
- "Pull Latest from GitHub"
- "Push to GitHub"

This makes the vault accessible to non-power-users like GMs who should not need to know command palette shortcuts.

### Medium-Priority Additions (Consider These)

#### Projects (by Marcus Olsson)
**What it does:** Provides table, board, calendar, and gallery views over folders of notes, reading frontmatter properties as columns.

**Why TFM needs it:** Could replace or supplement the Dataview-based `client-portfolio.md` dashboard with an interactive view where you can filter by GM, sort by risk level, or group by ESP -- all without writing Dataview queries. Each "row" links to the actual client file. More user-friendly for team members who will not learn Dataview syntax.

**Trade-off:** Overlaps with Dataview for some use cases. Best deployed for the client portfolio view specifically, while keeping Dataview for ad-hoc queries.

#### Hover Editor
**What it does:** Pop out any note into a floating window by hovering over a link and pressing a hotkey.

**Why TFM needs it:** During bi-weekly prep, a GM needs the client intelligence file, the last Friday report, and the meeting notes template all visible. Hover Editor lets you pop out reference files without cluttering the tab bar. Particularly useful on single-monitor setups.

#### DB Folder
**What it does:** Creates database-like views (similar to Notion databases) from folders of notes.

**Why TFM needs it:** The `clients/` folder is a natural database -- 25 subfolders, each with structured frontmatter (status, gm, esp, cpl_target, risk_level). DB Folder would render this as a filterable, sortable table without Dataview syntax. More intuitive for non-technical team members.

**Trade-off:** Some overlap with Projects plugin. Pick one or the other, not both. Projects is more mature; DB Folder is more Notion-like.

### Lower-Priority (Nice-to-Have)

#### Excalidraw
**What it does:** Embedded whiteboard/diagramming inside Obsidian notes.

**Why TFM needs it (maybe):** Could be useful for mapping creative strategy flows, client onboarding processes, or the media buying decision tree. However, TFM's core work is text-heavy (copy, reports, intelligence files), not diagram-heavy. Install only if visual mapping becomes a real need.

**Note:** Excalidraw files are large (JSON blobs) and would bloat the git repo. If installed, add `*.excalidraw` to .gitignore or keep them in a local-only folder.

#### Breadcrumbs
**What it does:** Adds hierarchy navigation (parent/child/sibling) based on frontmatter or folder structure.

**Why TFM needs it (maybe):** Could surface "this client belongs to this GM, who belongs to this team" relationships. But the vault's folder structure already handles this naturally (`clients/[name]/`). Skip unless the vault gets significantly more nested.

#### Obsidian Columns
**What it does:** Side-by-side column layouts in notes.

**Why TFM needs it (maybe):** Could improve the bi-weekly prep template (performance table on left, action items on right). But Obsidian's native split-pane works fine for this. Low priority.

### Do NOT Install

- **Bases** (already enabled as core plugin in Obsidian 1.8+) -- provides native database views. Try this before installing DB Folder or Projects.
- **Calendar** alternatives -- the current Calendar plugin works fine with Periodic Notes.
- **Obsidian Publish** -- TFM does not need a public-facing site from this vault.

## 3. GitHub Improvements

### 3.1 GitHub Actions / Workflows to Add

#### Markdown Lint CI
Create `.github/workflows/lint.yml` to run `markdownlint` on every push. Catches broken frontmatter, malformed tables, and inconsistent formatting before they reach the team.

```yaml
name: Lint Markdown
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: DavidAnson/markdownlint-cli2-action@v19
        with:
          globs: '**/*.md'
          config: '.markdownlint.json'
```

Pair with a `.markdownlint.json` config that allows the patterns TFM uses (inline HTML for tables, long lines in Dataview blocks, etc.).

#### Frontmatter Validation CI
A custom action that checks every file in `clients/` has the required frontmatter fields: `client`, `slug`, `status`, `gm`, `media_buyer`, `esp`, `cpl_target`, `risk_level`, `sentiment`, `last_enriched`. This prevents incomplete client files from being committed.

```yaml
name: Validate Client Files
on:
  push:
    paths: ['clients/**/*.md']
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          for f in clients/*/*.md; do
            for field in client slug status gm media_buyer esp cpl_target risk_level; do
              grep -q "^${field}:" "$f" || echo "MISSING $field in $f"
            done
          done
```

#### Weekly Backup to a Separate Branch
Create a scheduled action that copies the vault state to a `backup/weekly` branch every Sunday. Provides a restore point independent of Obsidian Git's continuous commits.

### 3.2 Branch Strategy

**Current state:** Everyone on `main`, Obsidian Git auto-pushes every 10 minutes. This works for a solo user but will cause merge conflicts with 2+ people.

**Recommended approach for team rollout:**

| Scenario | Strategy |
|----------|----------|
| Jay only (current) | Single `main` branch, auto-push. Fine as-is. |
| Jay + 1-2 GMs | Keep `main`. Obsidian Git's merge strategy handles small conflicts. Increase auto-pull interval to 5 min. |
| 3+ editors | Switch to **per-person branches** (`jay`, `lays`, `rabii`) with PRs to `main`. Use GitHub Desktop for non-technical team members. Obsidian Git pushes to their branch; Jay or Sindy merges to main. |
| Full team (10) | Per-person branches + **branch protection on main** (require PR, require 1 review). Add a CODEOWNERS file so client file changes ping the assigned GM. |

**Action now:** Since the shared-workspace-research.md already recommends GitHub Desktop for team members, the per-person branch strategy aligns with that plan. Set up branch protection rules on main before adding the second contributor.

## 4. Obsidian + Claude Code Integration

### 4.1 Smart Connections Status

**Current state:** Partially configured. Issues found:
- `new_user: true` flag is still set -- the onboarding wizard was likely skipped or interrupted
- Using local `TaylorAI/bge-micro-v2` embedding model (very small, ~25M params). This is fine for a 100-file vault but produces lower-quality semantic matches than larger models
- Chat model set to Ollama but no Ollama endpoint configured -- Smart Chat will not work
- File/folder exclusions are minimal (only "Untitled" files excluded)

**Recommendations:**
1. Re-run Smart Connections setup. Clear the `new_user` flag by opening settings and completing the wizard.
2. The local embedding model is adequate for TFM's vault size. No need to switch to an API-based model unless you notice poor semantic results.
3. Either configure Ollama locally or switch Smart Chat to use Copilot Plus (which is already licensed). However, Smart Chat is largely redundant with Claude Code -- you are already getting better results from Claude Code's direct filesystem access.
4. Add folder exclusions for `.obsidian`, `scripts`, and `prompts` (these do not need semantic embeddings).

### 4.2 Copilot Plus Status

**Current state:** Active with Plus license. Using `copilot-plus-flash` model. No external API keys configured (relying entirely on Copilot's bundled models).

**Issue:** The `data.json` containing the license key is being committed to GitHub. Fix this immediately:

Add to `.gitignore`:
```
.obsidian/plugins/copilot/data.json
```

Then remove the tracked file:
```bash
git rm --cached .obsidian/plugins/copilot/data.json
```

### 4.3 Obsidian MCP Plugin

There is a community plugin called **Obsidian MCP** (or "MCP Bridge") that exposes the vault as an MCP server. This would let Claude Code (or any MCP client) read/write vault files through the plugin's API.

**Verdict: Not needed.** Claude Code already reads the vault directly from the filesystem. The MCP plugin adds a layer of indirection with no benefit. It is designed for cases where the AI cannot access the filesystem directly -- which is not TFM's situation.

### 4.4 Making the Vault More Claude-Friendly

The vault is already well-structured for Claude Code. Specific improvements:

1. **Enable Dataview JavaScript queries.** This lets Claude Code generate more complex dashboards. Currently disabled by default. In Dataview settings, enable "Enable JavaScript Queries" and "Enable Inline JavaScript Queries."

2. **Standardize frontmatter across all client files.** Some files have slightly different field names or missing fields. A frontmatter validation script (see GitHub Actions section) would enforce consistency.

3. **Add a `_index.md` to each client folder.** Currently, some client folders have 2-4 files (intelligence file, client-config, claude-chat-project, deep-enrichment). An index file that links them all would help both human navigation and Claude Code context gathering.

4. **Move `CLAUDE.md` instructions into more granular files.** The current CLAUDE.md is 160+ lines. Consider splitting:
   - `CLAUDE.md` -- top-level project identity and conventions (kept small)
   - `system/claude-instructions-clients.md` -- client-specific instructions
   - `system/claude-instructions-media-buying.md` -- media buying conventions
   This keeps CLAUDE.md lean while giving Claude Code targeted context via explicit file reads.

   **Counter-argument:** CLAUDE.md is auto-loaded by Claude Code on every session. Splitting it means some instructions only load when explicitly read. The current approach of one comprehensive file may be better for reliability.

5. **Create a `vault-manifest.md`** -- a single file listing every file in the vault with a one-line description. Claude Code can read this at session start to understand what is available without scanning the entire filesystem.

### 4.5 Templater + Claude Code Workflow

Templater's `enable_system_commands` is currently `false`. Enabling it would allow templates to run shell commands -- including calling Claude Code CLI from within Obsidian. For example, a template could:

1. Create a new client file from template
2. Run a shell command that invokes Claude Code to auto-populate the file from Day.ai or Slack data

This is powerful but risky (arbitrary shell execution). Only enable if Jay is the sole Templater user, or lock down which commands are allowed.
